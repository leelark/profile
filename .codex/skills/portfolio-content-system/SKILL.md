# Portfolio Content System

Make content evidence-led, consistent, public-safe, premium, and business-value focused. Position Leelark as an Enterprise Appian Solution Architect, Senior Appian Consultant, Technical Lead, AI Solution Designer, Plugin Developer, Product Builder, Workflow Automation Specialist, and Digital Transformation Consultant.

## Standard Project Structure

Every project, plugin, innovative project, and accelerator should include:

1. Portfolio Classification: Primary Category, Secondary Categories, Evidence Status, Evidence Basis
2. Hero Summary
3. Project Context
4. Problem
5. My Role
6. Users and Stakeholders
7. Solution Overview
8. Architecture and Technical Approach
9. Key Features
10. UX and Product Decisions
11. Business Value
12. Technologies and Appian Capabilities
13. Suggested Website Sections
14. Suggested Visuals
15. Final Portfolio Copy
16. Confirmation Needed

## Evidence Rules

Use:

- `Confirmed` when reviewed material directly supports the claim.
- `Partially Confirmed` when related evidence exists but ownership, final implementation, or public wording needs confirmation.
- `Needs Confirmation` when user requested inclusion but source support is incomplete.

Do not turn `Needs Confirmation` into public claims.

## Content Tone

Prefer:

- architecture decisions
- product thinking
- business value
- reusable patterns
- user outcomes
- Appian-specific detail
- AI and workflow design clarity

Avoid:

- generic task lists
- inflated metrics
- unverified claims
- raw internal names
- overclaiming ownership
